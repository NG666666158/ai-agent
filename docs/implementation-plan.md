# Orion Agent 当前实施计划

## 目标

基于当前已经具备的任务执行、会话管理、长期记忆和用户画像能力，把项目从“可运行原型”推进到“可维护、可治理、可继续产品化”的原型平台。

## 当前阶段判断

当前最值得优先投入的不是继续堆更多页面，而是收两件事情：

1. 统一文档与项目认知
2. 增强执行内核的恢复、重规划和上下文分层

原因是：

- 前端页面和接口已经比较完整
- 当前的主要风险不再是“没有功能”，而是“内核治理不够细”
- 如果不先把内核和认知收稳，后续多用户、权限和生产化会越来越难做

## 阶段拆分

### 阶段 1：文档统一与能力边界收口

交付目标：

- 重写 `README.md`
- 重写 `docs/current-architecture.md`
- 重写 `docs/product-roadmap.md`
- 明确唯一主实现和当前真实边界

验收标准：

- 文档可直接给新开发者阅读
- 文档中不再出现乱码、历史歧义和过时能力描述
- 文档和代码主链路保持一致

### 阶段 2：执行内核增强

交付目标：

- 增强 checkpoint 模型
- 细化重规划原因和恢复状态
- 显式建立上下文分层与预算
- 为后续结构化引用与多用户隔离打底

验收标准：

- 任务恢复信息更细
- 失败和重规划原因更清晰
- 上下文构建有明确层级和长度控制
- 不破坏现有流式输出、会话、记忆与画像链路

### 阶段 3：会话与记忆治理增强

交付目标：

- 结构化来源追溯
- 记忆与画像的更强可控能力
- 多轮续聊中的上下文解释能力

验收标准：

- 用户能看见回答如何使用历史来源
- 记忆和画像的修改、合并、归档更可审查
- 会话追溯链路更完整

### 阶段 4：用户体系、权限和生产化

交付目标：

- 引入用户隔离
- 引入角色权限与审计
- 补生产部署和运维能力

验收标准：

- 任务、会话、记忆、画像可按用户隔离
- 高风险操作有更完整权限流程
- 服务具备基础生产治理能力

## 当前推荐执行顺序

1. 先完成文档统一
2. 再完成执行内核增强第一轮
3. 然后补来源治理和记忆治理
4. 最后进入多用户和生产化

## 本轮正在执行的内容

本轮聚焦两件事：

1. 清理并重写文档为完整中文可读版
2. 输出新的执行内核改造清单，并开始实现：
   - checkpoint 增强
   - 重规划增强
   - 上下文分层增强
