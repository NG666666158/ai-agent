# 执行内核改造清单

## 目标

把当前执行内核从“能跑通主流程”提升到“可恢复、可解释、可继续演进”的状态，重点补齐失败治理、断点恢复和上下文分层三条能力线。

## 本轮已落地

### 1. Checkpoint 增强

- 任务检查点新增阶段字段 `phase`
- 新增最近完成步骤 `last_completed_step_id` / `last_completed_step_name`
- 新增恢复原因 `resume_reason`
- 新增上下文版本 `context_version`
- 新增失败统计与恢复信息
  - `failure_count`
  - `recovery_attempt`
  - `last_failure_category`
  - `last_failure_resolution`

### 2. 失败分类与恢复决策

系统现在会把失败类别映射到恢复策略，而不只是简单记错：

- `TOOL_TIMEOUT` / `NETWORK_ERROR` / `TOOL_UNAVAILABLE`
  - 恢复策略：`REPLAN_FROM_CHECKPOINT`
- `INTERNAL_ERROR`
  - 恢复策略：`RETRY_CURRENT_STEP`
- `PERMISSION_DENIED`
  - 恢复策略：`REQUIRE_USER_ACTION`
- `INPUT_ERROR` / `VALIDATION_ERROR`
  - 恢复策略：`FAIL_FAST`

### 3. 可恢复重规划流

当工具级失败不可直接继续时，系统会：

1. 记录失败分类与恢复策略
2. 写入重规划事件 `ReplanEvent`
3. 基于当前检查点重建计划
4. 按失败原因调整执行条件

当前已实现的典型策略：

- 联网检索失败后，自动切换为离线规划
- 保留最近一次失败类型和恢复策略，便于前端解释“系统刚才为什么这样处理”
- 在重规划前写入上下文构建说明，便于后续审计与调试

### 4. 当前步骤重试

对于内部异常等瞬时失败，系统会优先尝试“重试当前步骤”而不是整任务失败：

- 重试上限由 `AGENT_EXECUTION_RECOVERY_RETRIES` 控制
- 重试过程会输出 `recovery` 进度事件
- 成功恢复后仍保留最近一次失败痕迹，用于可追溯展示

## 已补测试

本轮新增或调整的测试覆盖了：

- 工具失败后仅标记失败，由运行时决定恢复策略
- 联网检索超时后触发重规划并成功完成
- 执行阶段内部异常后重试当前步骤并成功恢复
- API 与主任务流在内核增强后保持兼容

## 当前状态

执行内核已经具备第一层“恢复能力”，但还不是最终形态。当前更像是：

- 有结构化失败分类
- 有任务级恢复循环
- 有从检查点重规划的能力
- 有上下文版本和恢复痕迹

还没有做到的部分主要是更细粒度的恢复决策。

## 下一轮建议

### A. 失败分类继续细化

- 区分工具参数错误、工具实现错误、模型生成错误、反思失败
- 为每类失败补更明确的用户可见文案

### B. 恢复策略继续细化

- 支持“跳过失败步骤继续执行”
- 支持“只重建后半段计划”
- 支持“恢复到最近成功步骤而不是整段重跑”

### C. 上下文分层继续增强

- 增加上下文来源映射
- 增加压缩记录和裁剪记录
- 为回答引用、来源锚点、历史追溯提供底层支撑

### D. 前端展示联动

- 展示失败类型与恢复策略标签
- 展示“系统已自动重试 / 自动重规划”的状态提示
- 展示最近一次失败来自哪个步骤、哪类工具

## 验收标准

本轮完成后，系统至少满足：

- 可以看出任务失败发生在哪个阶段
- 可以看出系统选择了哪种恢复策略
- 可以区分“重试当前步骤”和“从检查点重规划”
- 恢复成功后仍能追溯最近一次失败痕迹
- 不破坏现有会话、记忆、画像、流式输出和 API 链路
