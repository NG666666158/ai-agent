# 企业级稳定性路线图 v1

## 1. 路线图定位

本文档用于指导 `Orion Agent` 从：

- 一个已经成形的可视化 Agent Web 原型平台

逐步演进为：

- 一个具备接近 OpenClaw / claw code 这类企业级稳定性思路的 Agent Runtime 内核

这里的“接近”非常重要。

它的含义不是：

- 把项目做成 OpenClaw 的克隆
- 把 Web 项目改成纯 CLI
- 一次性做成庞大平台

它真正的含义是：

- 学习成熟运行时项目的内核边界和稳定性方法
- 保留当前项目已经形成的 Web 可视化产品优势
- 以渐进方式把当前内核重构为更稳定、更可恢复、更可治理的主骨架

---

## 2. 我们要学什么，不学什么

### 2.1 要学的核心内核经验

OpenClaw / claw code 这类项目最值得学习的是：

- 唯一主执行入口
- runtime 与产品表现层解耦
- 执行节点注册表
- ToolPool 与权限上下文
- SessionStore 作为基础设施边界
- 更清晰的恢复、重试、重规划模型
- 更适合长期扩展的 contract / schema 思维

### 2.2 不直接照搬的部分

当前阶段不建议直接照搬：

- 纯 CLI 优先的产品形态
- 过早铺开远程、bridge、voice、vim 等外围模式
- 过早做大而全插件市场
- 为了靠近某个开源项目而牺牲当前 Web 控制台体验

---

## 3. 当前判断

当前项目已经具备：

- 唯一主执行入口持续收口的基础
- 可视化执行链路
- 流式回答与系统进度
- 会话、记忆、画像主链路
- 基础恢复、重试、重规划能力
- 基础测试体系
- 一轮对齐 claw code 思路的 R1-R10 重构成果

但距离企业级稳定性内核，仍有明显差距：

1. 恢复状态机还没有完全做硬。
2. 上下文预算、裁剪、压缩仍不够统一。
3. ToolPool 与权限上下文仍有继续收口空间。
4. 会话、记忆、画像治理还没有完全平台化。
5. 多用户、权限、审计、部署治理尚未进入完成态。

---

## 4. 总体推进原则

### 原则 1：先稳定，再平台化，再企业化

推进顺序必须是：

- `P0` 稳定性
- `P1` 平台化
- `P2` 企业化

### 原则 2：不推翻现有主产品形态

必须保留并继续增强：

- Web 工作台
- 中文体验
- 执行过程可视化
- 会话 / 记忆 / 画像的直接可见性
- 回答与来源映射能力

### 原则 3：不再引入第二套主链路

后续所有重构必须继续围绕现有唯一主实现和统一 schema 收口，不再允许出现：

- 双轨执行逻辑
- 前后端语义不一致
- 为页面临时效果增加一次性结构

---

## 5. P0：稳定性

P0 的目标不是增加新页面，而是保证：

- 出错后能恢复
- 状态有统一真源
- 上下文不会失控
- 核心行为能被测试兜住

### P0-1 唯一主链路继续收口

目标：

- 保证任务执行、恢复、引用、上下文装配都只有唯一真源

交付物：

- 主执行入口文档继续明确
- 遗留双轨或半废弃逻辑继续清理
- runtime schema 统一

### P0-2 RecoveryPolicy 与恢复状态机做实

目标：

- 让失败分类与恢复决策形成真正稳定的主路径

交付物：

- 失败分类到恢复策略映射
- 跳过失败步骤继续执行规则
- 仅重建后半段计划规则
- 恢复痕迹结构化输出

### P0-3 ContextBuilder 与上下文预算做实

目标：

- 让上下文分层成为独立、可解释、可测试的稳定能力

交付物：

- 上下文层预算
- 裁剪与压缩记录
- 当前输入、短期历史、长期记忆、画像、工具结果的统一装配说明

### P0-4 ExecutionRegistry 继续做实

目标：

- 让执行节点真正成为运行时统一语义中心

交付物：

- 节点 kind / title / display meta / status 的统一定义
- 时间线与恢复说明围绕统一节点元数据工作

### P0-5 稳定性回归测试补齐

目标：

- 把测试重心转到“是否稳定、是否回归”

交付物：

- 恢复回归测试
- 上下文构建测试
- ToolPool 元数据测试
- 节点契约测试

---

## 6. P1：平台化

P1 的目标是让项目从“一个 Agent 应用”成长为“一个具备基础平台能力的 Agent Runtime”。

### P1-1 ToolPool 平台化

目标：

- 工具不只是可调用，而是具备完整元数据和权限上下文

交付物：

- 工具类别
- 风险等级
- 超时 / 重试 / 展示标签
- 权限上下文

### P1-2 CitationMap 与来源解释平台化

目标：

- 引用映射不再依赖前端推断，而由后端稳定产出

交付物：

- 段落到来源的结构化映射
- 记忆 / 画像 / 会话消息 / 外部材料统一来源模型

### P1-3 SessionStore 平台化

目标：

- 会话不再只是页面功能，而成为稳定基础设施层

交付物：

- 会话存储边界
- 摘要刷新机制
- 分支续聊规则
- 会话追溯接口

### P1-4 Memory Governance

目标：

- 长期记忆从“能存能召回”升级到“可治理、可回滚、可解释”

交付物：

- 记忆类型
- 命中原因
- 多路召回
- rerank
- 版本记录与软删除

### P1-5 Profile Governance

目标：

- 用户画像从“自动积累结果”升级到“长期可控资产”

交付物：

- 偏好提取器
- 画像编辑
- 冲突合并
- 人工确认
- 跨会话画像注入治理

---

## 7. P2：企业化

P2 的目标是让项目具备组织级使用和生产部署基础。

### P2-1 多用户隔离

交付物：

- `user_id` 接入核心查询路径
- 会话 / 任务 / 记忆 / 画像按用户隔离

### P2-2 权限体系

交付物：

- 工具权限模型
- 高风险操作审批链
- 角色级或策略级权限边界

### P2-3 审计与留痕

交付物：

- 高风险操作审计
- 记忆编辑审计
- 画像合并审计
- 恢复与重规划审计

### P2-4 生产部署能力

交付物：

- 运行环境分层
- 健康检查细化
- 指标与日志说明
- 故障排查手册
- 备份恢复机制

---

## 8. 推荐推进顺序

建议按下面顺序推进：

### Step 1：收口稳定性主骨架

先完成：

- 唯一主链路继续收口
- RecoveryPolicy 与恢复状态机做实
- ContextBuilder 与预算控制做实
- ExecutionRegistry 继续收口

### Step 2：补平台中层

再完成：

- ToolPool
- CitationMap
- SessionStore
- Memory Governance
- Profile Governance

### Step 3：补企业底座

最后完成：

- 多用户隔离
- 权限体系
- 审计
- 生产部署与运行治理

---

## 9. 当前推荐下一阶段

如果现在只选一段继续往前做，建议优先选择：

> P0：恢复状态机 + 上下文分层 + ToolPool 权限上下文

原因是：

- 这是最接近企业级稳定性内核骨架的关键基础
- 做稳之后，平台中层和产品层都会更容易继续收口

---

## 10. 成败判断标准

这轮路线是否成功，不看页面数量，而看下面 6 个问题能否都回答“是”：

1. 是否只有唯一主链路？
2. 是否可以恢复而不是只能重跑？
3. 是否能解释系统为什么这样做？
4. 是否能通过自动化测试兜住核心行为？
5. 是否能治理会话、记忆和画像？
6. 是否为权限、用户和部署留出了稳定边界？

如果这 6 个问题都能逐步回答“是”，项目就已经不是“功能很多的原型”，而是在真正走向企业级稳定的 Agent 平台内核。
