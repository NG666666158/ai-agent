# Orion Agent 后续开发清单

## 1. 清单定位

这份清单是当前项目后续开发的统一执行入口。

它以以下目标为中心：

> 保留你最开始想要的“可视化 Agent Web 工作台”，同时把执行内核逐步重构为接近 OpenClaw / claw code 这类企业级稳定性思路的 Agent Runtime。

因此，清单排序不是按“哪块看起来最酷”，而是按“哪块最影响系统稳定性、可治理性和后续扩展能力”来排。

---

## 2. P0：执行内核稳定性

P0 关注的是：

- 唯一主执行链路是否稳定
- 出错后是否能恢复
- 上下文是否可控
- 执行痕迹是否可解释

### P0-1 恢复状态机继续做实

目标：

- 把当前已有的失败恢复流真正做成稳定主路径，而不是“有能力但边界不够硬”

预计工作：

- 细化 checkpoint 快照结构
- 明确失败分类到恢复策略的映射
- 做实“跳过失败步骤继续执行”
- 做实“仅重建后半段计划”
- 统一恢复痕迹输出与前端展示

预计交付物：

- 更完整的 `TaskCheckpoint` 模型
- 更清晰的 `RecoveryPolicy` 行为约束
- 恢复流回归测试
- 恢复链路说明文档

### P0-2 上下文分层与预算控制做实

目标：

- 把当前上下文分层从“结构已存在”推进到“预算、裁剪、压缩、解释都稳定”

预计工作：

- 为系统提示、当前输入、短期历史、长期记忆、用户画像、工具结果建立预算
- 记录裁剪原因和压缩结果
- 增强上下文调试信息
- 增强多轮续聊中的摘要与增量压缩

预计交付物：

- 更完整的 `ContextLayer` / `ContextBuilder` 输出
- 预算与裁剪记录结构
- 上下文构建测试与回归测试

### P0-3 ToolPool 与权限上下文收口

目标：

- 把当前工具系统继续推进为带元数据、权限和风险标签的工具池

预计工作：

- 完善工具类别、展示标签、超时、重试、风险级别
- 打通工具元数据到前端展示
- 明确高风险工具的确认上下文
- 为后续权限矩阵保留稳定边界

预计交付物：

- ToolPool 元数据模型增强
- 工具权限上下文字段
- 工具调用展示统一

### P0-4 引用映射与恢复痕迹后端统一

目标：

- 让来源解释和恢复解释真正依赖后端结构化产物，而不是前端猜测

预计工作：

- 后端 citation map 继续收口
- 恢复原因、恢复策略、恢复次数、恢复起点结构化输出
- 统一段落引用与任务追溯字段

预计交付物：

- 稳定的 citation / recovery schema
- 前端引用锚点和恢复详情升级

---

## 3. P1：治理与平台中层

P1 关注的是：

- 会话、记忆、画像是否能长期治理
- 系统是否能从“原型应用”迈向“可扩展平台”

### P1-1 会话层追溯与治理增强

目标：

- 让 SessionStore 真正承担长期会话中枢职责

预计工作：

- 会话摘要刷新策略收口
- 分支续聊规则明确
- 会话到任务、记忆、画像的追溯链路增强
- 新旧会话关系展示增强

预计交付物：

- 会话追溯数据结构增强
- SessionStore 文档与测试补强

### P1-2 记忆治理闭环增强

目标：

- 让长期记忆从“能召回”变成“可治理、可纠错、可解释”

预计工作：

- 记忆类型治理
- 命中原因治理
- 版本记录
- 软删除与回滚
- 低质量记忆清理策略

预计交付物：

- 记忆治理字段扩展
- 记忆管理页增强
- RAG 评估与回归用例

### P1-3 画像治理闭环增强

目标：

- 让用户画像成为长期稳定资产，而不是自动堆积结果

预计工作：

- 偏好提取策略增强
- 画像确认与冲突审查
- 人工编辑优先级规则
- 画像来源说明增强

预计交付物：

- 画像治理状态字段
- 画像编辑与审查体验增强
- 画像链路测试补齐

### P1-4 控制台体验统一收尾

目标：

- 让当前控制台真正达到“稳定可演示”的统一状态

预计工作：

- 中文文案全仓巡检
- Markdown 复杂内容展示精修
- 思考过程折叠体验优化
- 异常态和空态统一

预计交付物：

- 控制台 UI 收尾版
- 文案统一记录

---

## 4. P2：企业化底座

P2 关注的是：

- 从单用户原型走向可组织化使用
- 从本地演示走向稳定部署

### P2-1 多用户隔离

目标：

- 建立真实用户边界

预计工作：

- 引入 `user_id`
- 会话、任务、记忆、画像按用户归属
- 查询路径按用户隔离

预计交付物：

- 用户归属字段接入核心查询
- 多用户隔离测试

### P2-2 权限体系与审计

目标：

- 把“高风险确认”升级为完整权限与审计能力

预计工作：

- 角色或策略级权限模型
- 高风险工具权限矩阵
- 审批与拒绝原因留痕
- 恢复、重规划、记忆编辑的审计记录

预计交付物：

- 权限模型文档
- 审计日志结构
- 审计接口与页面预留

### P2-3 生产部署与运行治理

目标：

- 建立基础生产化能力

预计工作：

- 环境配置分层
- 健康检查细化
- 指标与日志说明补齐
- 备份恢复与故障排查流程

预计交付物：

- 中文生产部署说明
- 指标与运维手册
- 最小备份恢复方案

---

## 5. 推荐执行顺序

建议后续按下面顺序推进：

1. `P0-1` 恢复状态机继续做实
2. `P0-2` 上下文分层与预算控制做实
3. `P0-3` ToolPool 与权限上下文收口
4. `P0-4` 引用映射与恢复痕迹后端统一
5. `P1-1` 会话层追溯与治理增强
6. `P1-2` 记忆治理闭环增强
7. `P1-3` 画像治理闭环增强
8. `P1-4` 控制台体验统一收尾
9. `P2-1` 多用户隔离
10. `P2-2` 权限体系与审计
11. `P2-3` 生产部署与运行治理

---

## 6. 当前推荐下一步

如果现在只优先选一项继续做，建议是：

### 当前下一步推荐：`P0-1 恢复状态机继续做实`

原因：

- 这是把当前内核继续向 OpenClaw / claw code 稳定性骨架推进的核心一步
- 当前很多体验问题，最终都能追溯到恢复流和上下文治理是否稳定
- 这一步做稳后，引用、记忆、会话、权限体系都会更容易收口

---

## 7. 统一结论

后续开发不应再按“哪里不爽就补哪里”的方式推进，而应统一按以下主线前进：

> 先把执行内核做稳，再把会话、记忆、画像和引用治理做深，最后进入多用户、权限和生产化阶段。
