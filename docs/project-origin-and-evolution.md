# Orion Agent 项目起源与演进记录

## 1. 文档目的

本文档用于固定两件事情：

- 项目最初的真实需求和目标，不因为后续多轮重构而丢失初心
- 项目从 MVP 到当前阶段的主要演进过程，作为后续继续重构的背景基线

本文档优先回答三个问题：

1. 这个项目最开始到底想做什么？
2. 现在已经做到什么程度了？
3. 为什么后续要把内核继续往 OpenClaw / claw code 这类企业级稳定性思路上推进？

---

## 2. 最初需求

### 2.1 最初产品目标

项目最开始的核心目标，不是单纯做一个“能回答问题的聊天框”，而是做一个：

> 具备可视化执行链路、可追溯思考过程、可持续演进内核的中文 AI Agent Web 项目

这里面包含了几个非常明确的初始诉求：

- 前端要像一个真正的 Agent 产品，而不是普通聊天页面
- 用户输入问题后，系统不只返回最终答案，还要展示中间执行过程
- 思考过程要能折叠，不影响最终回答阅读
- 会话、记忆、画像、来源、恢复痕迹都要可视化
- 页面主要面向中文使用场景，文案和体验尽量统一为中文

### 2.2 最初前端形态诉求

最初对前端体验的目标，已经比较接近典型的 Agent 控制台产品：

- 左侧：可收起的历史会话栏
- 右侧：当前对话窗口与当前会话记录
- 回答区：只突出最终 LLM 输出结果
- 思考区：放在回答下方，可折叠展示
- 思考区内容：展示所有执行流程，包括 query 改写、向量检索、多路召回、提示词拼接、工具调用、恢复过程等
- 回答区需要支持 Markdown，不能停留在纯文本展示

### 2.3 最初后端与内核诉求

最初项目就不是要停留在“前端做个壳、后端拼个 API”，而是希望逐步做成一个：

- 有执行主链路的 Agent 系统
- 有会话层、记忆层、画像层的长期交互系统
- 有测试体系和可回归能力的工程项目
- 最终能把内核稳定性打磨到接近 OpenClaw / claw code 这种企业级产品思路

最重要的一条长期目标可以归纳为：

> 保留当前 Web 可视化 Agent 工作台的产品优势，同时把执行内核逐步重构为更稳定、更可恢复、更可扩展、更可治理的企业级运行时内核。

---

## 3. 演进过程回顾

### 3.1 早期阶段：单任务 Agent MVP

早期阶段的项目重点在于跑通最基本的主链路：

- 接收用户任务
- 调用 LLM
- 生成回答
- 展示简单结果

这个阶段完成的是“能跑”，但还没有完成：

- 会话持续性
- 记忆复用
- 画像注入
- 执行过程可视化
- 失败恢复
- 稳定性治理

### 3.2 第二阶段：可视化 Agent 工作台成形

随后项目逐步进入“产品雏形建立”阶段，主要变化包括：

- 前端开始具备控制台形态
- 回答改为流式正文输出
- 系统进度区和回答生成区分开
- 思考过程、步骤时间线、来源提示、引用脚注开始落地
- 会话列表、任务详情、记忆页、画像页逐步补齐

这个阶段的意义在于：

- 项目已经不再只是一个后端实验脚本
- 已经成为一个可演示、可测试、可继续迭代的 Agent 工作台

### 3.3 第三阶段：会话、记忆、画像主链路补齐

在产品体验成形后，系统开始补长期交互能力：

- `chat session`
- 会话历史与上下文压缩
- 长期记忆召回与编辑
- 用户画像提取、跨会话注入、画像展示与冲突合并
- 来源提示与回答引用锚点

这一阶段解决的关键问题是：

- 系统不再每开新会话就完全“失忆”
- 回答开始具备“为什么这样回答”的来源解释能力

### 3.4 第四阶段：向平台化运行时内核演进

在产品层基础形成后，项目开始进入更深层的内核重构阶段，重点转向：

- 唯一主实现收口
- 上下文分层
- 执行节点标准化
- 恢复策略抽离
- 引用映射后端化
- 会话与记忆边界标准化

最近一轮对齐 claw code 思路的重构，已经完成了 R1-R10 主线：

- `US-R1`：统一执行节点主实现
- `US-R2`：拆出 `ContextBuilder`
- `US-R3`：引入 `ExecutionRegistry`
- `US-R4`：拆出 `RecoveryPolicy`
- `US-R5`：预留 ToolPool / Citation 接口
- `US-R6`：做实 ToolPool 元数据与权限管线
- `US-R7`：后端 CitationMap 标准化
- `US-R8`：抽出 SessionStore 边界
- `US-R9`：上下文契约与 Trace 元数据增强
- `US-R10`：记忆与画像治理增强

这意味着：

- 当前项目已经不只是“功能堆叠”
- 而是在逐步形成可以继续平台化演进的内核骨架

---

## 4. 当前真实阶段

截至 `2026-04-18`，项目当前更适合定义为：

> 一个已经完成主体产品形态、并且正在把执行内核继续推进到企业级稳定性方向的 Agent 原型平台

当前已经具备：

- 可视化 Agent 控制台
- 流式回答与系统进度
- 会话层、记忆层、画像层
- 引用与来源追溯
- 基础恢复、重试、重规划能力
- 基础测试体系
- 一轮较完整的平台化内核重构成果

当前仍然明显不足：

- 多用户体系
- 权限矩阵与审计
- 完整的生产部署与监控治理
- 更成熟的恢复状态机
- 更稳定的上下文预算与压缩策略

---

## 5. 为什么后续要对齐 OpenClaw / claw code 思路

这里的“对齐”不是抄界面，也不是把项目改成纯 CLI 工具，而是学习它们更成熟的内核思想：

- 唯一主执行入口
- 明确的 runtime 边界
- 执行节点注册表
- 工具池与权限上下文
- 会话存储边界
- 更稳定的恢复与重规划模型
- 更适合长期维护和扩展的契约设计

需要保留的则是当前项目自己的优势：

- Web 工作台
- 中文产品体验
- 可视化执行链路
- 会话、记忆、画像的直接可见性
- 回答结果、来源和恢复痕迹的统一展示

所以后续路线不是“变成 OpenClaw”，而是：

> 在保留 Web Agent 产品体验的前提下，把内核稳定性、边界清晰度、可恢复性和可治理能力逐步提升到接近企业级 Agent Runtime 的水平。

---

## 6. 当前统一目标

从现在开始，项目的统一目标可以表述为：

### 产品目标

做一个中文、可视化、可追溯、可持续演进的 Agent 工作台。

### 技术目标

把当前执行内核继续重构为：

- 唯一主实现
- 可恢复
- 可重规划
- 可分层上下文治理
- 可解释来源映射
- 可治理会话、记忆、画像
- 可继续扩展到多用户、权限和生产环境

### 长期目标

让项目最终具备：

- 接近 OpenClaw / claw code 这类企业级 Agent Runtime 的稳定性骨架
- 但保持当前 Web 可视化产品形态，而不是牺牲产品体验去换“仅内核强”

---

## 7. 后续开发的约束

为避免后续开发再次跑偏，应该统一遵守以下约束：

1. 不再新造第二套主执行链路。
2. 不再把前端体验优化和内核真相分离成两套语义。
3. 优先收口恢复流、上下文分层、ToolPool、会话边界和治理边界。
4. 新能力必须尽量挂到现有统一主结构上，而不是继续增加临时字段和临时流程。
5. 文档必须持续同步，不能再让路线图落后于代码。

---

## 8. 结论

一句话概括项目起点与现在的关系：

> 这个项目最初想做的是一个“前端可视化、执行过程可追溯、能持续演进的中文 Agent 工作台”；而现在它已经完成了原型主体建设，接下来最核心的任务，是把内核逐步打造成接近 OpenClaw / claw code 这种企业级稳定性思路的 Agent Runtime，同时保留当前 Web 产品形态和交互优势。
