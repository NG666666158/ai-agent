# AI Agent MVP 核心功能测试规范 (Test Specification)

## 0. 文档概述

本文件旨在定义 AI Agent MVP 版本的测试范围、用例逻辑及验收标准。测试重点在于验证 **“感知-规划-行动”** 链路的准确性与鲁棒性。

------

## 1. 单元测试 (Unit Testing)：模块级可用性

**目标**：确保各独立组件在隔离状态下逻辑 100% 正确。

| **模块名称**               | **测试重点**                                              | **边界场景 (Edge Cases)**                            |
| -------------------------- | --------------------------------------------------------- | ---------------------------------------------------- |
| **解析器 (Parser)**        | 结构化提取用户意图、参数识别                              | 空输入、超长文本（>10k tokens）、乱码/非语义输入     |
| **规划器 (Planner)**       | 任务拆解的逻辑合理性、步骤依赖关系                        | 单步任务、循环任务依赖、模糊意图（无法拆解时的反馈） |
| **工具调用 (Tool Caller)** | 参数映射准确性、类型校验                                  | 缺失参数调用、参数类型错误（如 String 传 Integer）   |
| **状态机 (State Machine)** | 状态流转（Idle -> Planning -> Executing -> Success/Fail） | 非法状态跳转、长时间运行后的状态挂起                 |

------

## 2. 集成测试 (Integration Testing)：链路协同

**目标**：验证全链路 `任务解析 → 规划 → 执行 → 工具调用 → 输出` 的数据流转。

- **典型测试流**：
  1. **输入**：用户请求“读取 `data.txt` 并总结核心观点”。
  2. **解析**：提取 Action=Read, Target=data.txt; Action=Summarize.
  3. **规划**：Step 1: File_Read -> Step 2: Text_Summarize -> Step 3: Output.
  4. **执行/调用**：验证 `File_Read` 的输出是否准确作为 `Text_Summarize` 的输入。
  5. **输出**：验证最终生成的 Markdown 格式是否包含总结内容。
- **验证点**：模块间无 Context 丢失，Token 传递未超出限制。

------

## 3. 场景化黑盒测试 (Scenario-based Testing)

**目标**：模拟真实用户 persona，验证 20+ 典型闭环场景。

### A. 开发者场景 (Developer Persona)

- **Case 01**: “分析这段 Python 代码的逻辑漏洞，给出修复代码，并保存为修复文档。”
- **Case 02**: “从指定的 GitHub Repo 链接中提取 README 内容，生成技术选型对比表。”

### B. 学生/研究员场景 (Academic Persona)

- **Case 03**: “上传 PDF 论文，提取其实验设置（Datasets, Metrics）并生成报告大纲。”
- **Case 04**: “搜索关于‘大语言模型幻觉’的最新三篇论文，对比其解决方案。”

### C. 办公用户场景 (Office Persona)

- **Case 05**: “整理以下会议记录（Paste Text），提取 Action Items 和负责人，生成邮件模板。”
- **Case 06**: “将本地文件夹中的 5 个文本文件合并，提取关键词并生成 Markdown 摘要。”

**验收指标**：闭环成功率 (Success Rate) ≥ 70%。

------

## 4. 异常场景与鲁棒性测试 (Robustness Testing)

**目标**：验证 Agent 在极端情况下的“不崩溃、可回溯”能力。

| **异常类型** | **模拟手段**                                           | **预期行为**                                       |
| ------------ | ------------------------------------------------------ | -------------------------------------------------- |
| **工具故障** | 模拟网络断开、文件权限被拒绝、API Key 失效             | 报错不卡死，清晰提示“工具 [X] 调用失败，原因：[Y]” |
| **任务变更** | 在规划完成后、执行中途，输入“停止，改为生成 Word 格式” | 触发重规划 (Re-planning)，终止当前步骤，更新目标   |
| **长链压力** | 输入需要拆解为 10 个以上步骤的复杂任务                 | 状态机流转稳定，无内存泄漏，能够持续追踪上下文     |
| **超时处理** | 设置工具调用响应时间 > 30s                             | 自动触发超时重试机制或优雅退出并记录日志           |

------

## 5. 人工验收测试 (Manual Acceptance)

**目标**：对齐 MVP 成功标准，确保产品体感。

- **可视化要求**：开发者必须能通过控制台或 UI 实时看到 `Thought -> Action -> Observation` 的链条。
- **可溯源性**：出错时，必须有 Trace ID 或详细 Log 记录哪一步规划出现了偏差。
- **性能基准**：
  - 核心工具（总结、读取）调用成功率 ≥ 90%。
  - 简单任务响应延迟（从输入到首个 Action 执行）≤ 3s。
