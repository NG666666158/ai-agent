# PRD：Orion Agent 企业级可追溯工作台演进

## 0. 文档说明

本文档基于当前项目代码状态整理，目标不是从零规划一个新项目，而是为现有 `Orion Agent` 原型提供下一阶段的可执行产品需求说明。

本 PRD 采用以下默认假设：

- 当前项目已经具备基础的任务执行、流式回答、会话、记忆、用户画像、测试脚本和中文控制台。
- 下一阶段的重点不是继续堆叠零散功能，而是统一主实现、增强执行内核、补齐可追溯能力，并让前端真正呈现 Agent 全流程。
- 目标读者包含后续开发者、测试人员以及可执行 story 的自动化 agent，因此要求描述明确、验收标准可验证。

## 1. Introduction / Overview

`Orion Agent` 当前已经从单任务演示升级为带有会话、长期记忆、用户画像和基础可观测能力的 Agent 原型，但整体仍存在三个核心问题：

1. 执行主链路仍然分散，前后端对“Agent 到底做了什么”的表达不统一。
2. RAG、记忆、画像、会话虽然都已存在，但可追溯、可解释、可控程度还不够稳定。
3. 项目文档、工程结构和自动化执行粒度还未整理成适合持续迭代和自动化开发的状态。

本阶段需要把项目从“功能已经不少的原型”推进到“结构更稳、流程可恢复、信息可追溯、界面更像成熟 Agent 产品”的状态。

## 2. Goals

- 明确唯一主实现链路，减少双轨逻辑和重复状态表达。
- 将 Agent 的执行过程统一为可视化、可追溯、可恢复的执行时间线。
- 增强 RAG 链路，使召回结果具备类型、分数、命中原因和最小 rerank 能力。
- 强化跨会话记忆与用户画像注入，使新会话可以稳定继承用户偏好与历史事实。
- 提升前端交互体验，使用户默认只看回答正文，需要时可以折叠查看完整思考流程。
- 补齐更适合自动化迭代的文档、story 拆分和执行清单。

## 3. User Stories

### US-001：统一执行主链路数据模型
**Description:** 作为开发者，我希望后端把解析、召回、规划、工具调用、恢复和结果复核统一成一套执行节点结构，这样前端和测试都能依赖唯一主实现。

**Acceptance Criteria:**
- [ ] `TaskResponse` 输出统一的执行节点字段，用于表达解析、召回、规划、执行、恢复、复核和回答生成
- [ ] 旧有分散字段仍可兼容，但前端主展示不再依赖多套重复结构
- [ ] 每个执行节点至少包含标题、状态、摘要、时间戳和可展开 artifact
- [ ] 单元测试覆盖执行节点构建逻辑
- [ ] Typecheck / 后端测试通过

### US-002：细化执行阶段与恢复痕迹
**Description:** 作为使用者，我希望在任务执行时看到更细颗粒的阶段变化，并在失败时知道失败类型、恢复策略、恢复次数和恢复起点。

**Acceptance Criteria:**
- [ ] 系统阶段至少覆盖输入解析、约束整理、记忆检索、多路召回、Prompt 拼接、计划生成、步骤执行、回答生成、结果复核
- [ ] 失败类型包含工具失败、评审失败、上下文问题、权限确认、未知异常等可区分分类
- [ ] 恢复事件包含恢复策略、恢复次数、从哪一步恢复、是否跳过失败步骤
- [ ] 前端可以直接展示恢复痕迹
- [ ] Typecheck passes

### US-003：实现可恢复的重规划流
**Description:** 作为系统维护者，我希望执行失败后不总是从头重跑，而是可以跳过失败步骤或仅重建后半段计划，以提升稳定性和执行效率。

**Acceptance Criteria:**
- [ ] 支持“跳过失败步骤继续执行”的恢复策略
- [ ] 支持“仅重建后半段计划”的恢复策略
- [ ] 恢复逻辑记录到 checkpoint / replan 历史中
- [ ] 失败恢复相关测试覆盖正常恢复与恢复失败场景
- [ ] Tests pass

### US-004：增强 RAG 元数据与最小混合召回
**Description:** 作为系统开发者，我希望每条记忆都带有类型字段、召回分数和命中原因，并支持最小混合召回加 rerank，这样回答依据会更稳定。

**Acceptance Criteria:**
- [ ] 长期记忆记录包含 `memory_type`
- [ ] 检索结果返回 `retrieval_score`、`retrieval_reason`、`retrieval_channels`
- [ ] 至少同时支持向量召回和规则/关键词辅助召回
- [ ] 对候选结果进行最小 rerank，输出最终排序
- [ ] 针对语义记忆召回链路补齐测试
- [ ] Tests pass

### US-005：补齐跨会话记忆与用户画像注入
**Description:** 作为用户，我希望在新会话中仍能被系统记住偏好、背景和长期事实，而不是每个窗口都重新开始。

**Acceptance Criteria:**
- [ ] 用户偏好提取器可以从消息中抽取稳定偏好事实
- [ ] 用户画像以结构化事实形式存储，并支持跨会话注入
- [ ] 新会话发起任务时自动带入画像和历史偏好摘要
- [ ] 回答中可提示哪些内容来自画像、记忆或历史会话
- [ ] Tests pass

### US-006：支持画像编辑与冲突合并
**Description:** 作为运营或测试人员，我希望可以手工修正用户画像并处理冲突事实，以保证长期记忆不越积越乱。

**Acceptance Criteria:**
- [ ] 提供画像事实编辑能力
- [ ] 支持把冲突画像事实合并到目标事实
- [ ] 保留被合并事实与目标事实的关系信息
- [ ] 前端可查看画像状态和来源
- [ ] Verify in browser using dev-browser skill
- [ ] Typecheck passes

### US-007：重构控制台为标准 Agent 对话布局
**Description:** 作为终端用户，我希望控制台像常见 Agent 产品一样，左侧管理历史会话，右侧专注当前对话和回答，思考过程可折叠查看。

**Acceptance Criteria:**
- [ ] 左侧存在可收起的会话列表
- [ ] 右侧存在当前对话窗口、消息流和输入区
- [ ] 思考过程默认折叠在回答下方，不抢占正文注意力
- [ ] 回答区和系统进度区语义明确、中文统一
- [ ] Verify in browser using dev-browser skill
- [ ] Typecheck passes

### US-008：升级前端执行过程可视化
**Description:** 作为用户，我希望看到完整的 Agent 执行流程，包括 query 改写、检索、召回、Prompt 拼接和工具调用，并能展开查看细节。

**Acceptance Criteria:**
- [ ] 思考过程以时间线或节点流形式展示
- [ ] 每个节点支持展示摘要与可展开 artifact
- [ ] 节点至少覆盖 query 改写、向量检索、多路召回、Prompt 拼接、工具调用、恢复、回答生成
- [ ] 运行中节点具备明显状态样式
- [ ] Verify in browser using dev-browser skill
- [ ] Typecheck passes

### US-009：升级回答区 Markdown 与来源锚点
**Description:** 作为用户，我希望最终回答是正常排版的 Markdown，同时能从段落跳到来源脚注，知道某段话具体引用了哪条记忆或画像。

**Acceptance Criteria:**
- [ ] 回答区正确渲染 Markdown 标题、列表、代码块、表格和引用
- [ ] 支持段落内来源锚点或脚注跳转
- [ ] 每条来源脚注能定位到记忆、画像或会话来源
- [ ] 提供复制回答功能
- [ ] Verify in browser using dev-browser skill
- [ ] Typecheck passes

### US-010：清理中文文档并统一项目认知
**Description:** 作为项目维护者，我希望 README、架构文档、路线图和测试报告模板统一为完整中文可读版，以降低后续维护成本。

**Acceptance Criteria:**
- [ ] README 改为完整中文说明
- [ ] 架构文档、路线图、开发清单和测试报告模板互相引用且术语一致
- [ ] 清理乱码或过时内容
- [ ] 文档明确当前完成度、边界和后续优先级
- [ ] Markdown 预览可读

### US-011：补强多用户隔离与权限基础
**Description:** 作为平台管理员，我希望后续能够将任务、会话、记忆和画像按用户隔离，并为高风险操作打下权限基础。

**Acceptance Criteria:**
- [ ] 核心数据模型预留 `user_id` 或等价归属字段
- [ ] 高风险工具调用具备权限级别定义
- [ ] 任务、会话、记忆、画像可按用户维度查询和过滤
- [ ] 明确当前只做基础模型与接口预留，不强推完整登录体系
- [ ] Tests pass

### US-012：补齐生产部署与运行观测基础
**Description:** 作为部署维护者，我希望项目具备更稳定的部署说明、指标输出和基础恢复建议，为后续生产化做准备。

**Acceptance Criteria:**
- [ ] 文档中明确本地、容器、生产部署差异
- [ ] 系统健康信息可区分 LLM、Embedding、Vector、Search 等运行状态
- [ ] 保留基础 Prometheus 指标输出说明
- [ ] 提供最小故障排查指引
- [ ] Markdown 文档可读

## 4. Functional Requirements

- FR-1：系统必须输出统一的 `execution_nodes` 结构，作为 Agent 执行全流程的主展示数据源。
- FR-2：系统必须将解析、上下文构建、记忆召回、工具调用、恢复与答案生成映射为可追溯节点。
- FR-3：系统必须记录失败分类、恢复策略、恢复次数和恢复起点。
- FR-4：系统必须支持可恢复的重规划流，而不是每次都从头执行。
- FR-5：长期记忆必须包含记忆类型、召回分数、命中原因和命中通道。
- FR-6：系统必须支持最小混合召回与 rerank，以提升记忆命中稳定性。
- FR-7：系统必须支持跨会话注入用户画像与稳定偏好。
- FR-8：前端必须使用中文界面，并将回答区与系统思考区区分展示。
- FR-9：前端必须为执行链路提供可折叠、可展开的细节展示。
- FR-10：回答正文必须支持稳定 Markdown 渲染与来源锚点跳转。
- FR-11：项目文档必须统一为中文可读版，并反映当前真实开发状态。
- FR-12：核心数据模型必须为多用户隔离和权限体系预留扩展空间。

## 5. Non-Goals

- 本阶段不建设完整商业化 SaaS 后台。
- 本阶段不强制接入真实登录注册、账单、团队协作和 RBAC 全套体系。
- 本阶段不追求复杂工作流编排平台或可视化拖拽编排。
- 本阶段不引入重量级第三方 Agent 框架替代当前原生实现。
- 本阶段不建设高复杂度数据治理后台，仅做必要的记忆/画像管理能力。

## 6. Design Considerations

- 前端默认聚焦“最终回答”，思考过程作为可折叠二级区域。
- 左侧会话、右侧当前对话的布局应贴近常见 Agent 产品，降低学习成本。
- 执行时间线要比日志列表更结构化，节点命名需要中文化且面向用户可理解。
- 来源提示既要告诉用户“用了什么”，也要支持定位“这段话具体来自哪条记忆/画像/历史消息”。

## 7. Technical Considerations

- 后端仍以 Python + FastAPI + 原生 Agent 流程为主，不引入 LangChain。
- 前端继续使用 React + TypeScript + Vite。
- 测试框架以后端 `unittest` 为主，前端优先保证构建可用和关键交互可回归。
- 现有接口较多，重构时要优先保证接口兼容，再逐步将前端切换到统一主实现。
- 所有 story 尽量控制在“一次聚焦开发可完成”的粒度，方便后续自动化执行或 AI 辅助开发。

## 8. Success Metrics

- 用户在主界面能清楚区分“回答正文”和“系统执行过程”。
- 新会话中可以稳定继承用户偏好，不再出现明显“忘记用户”的情况。
- 失败恢复不再只靠整任务重跑，至少支持局部恢复策略。
- 前端能稳定展示 Agent 全流程关键节点，而不是零散进度文本。
- README、架构文档和路线图能够帮助新开发者在 10 分钟内建立项目认知。

## 9. Open Questions

- 多用户隔离最终是通过本地 user_id、账号体系还是外部身份系统接入，尚需后续确定。
- RAG 的 rerank 后续是否要升级为独立模型或外部服务，当前只规划最小实现。
- 前端是否要进一步拆分出独立“任务详情页”和“执行追踪页”，当前可先在控制台与任务页共存。
