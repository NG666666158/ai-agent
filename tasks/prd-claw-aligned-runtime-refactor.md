# PRD：对齐 Claw Code 的运行时重构

## 1. Overview

当前 `Orion Agent` 已经具备可运行的 Web Agent 工作台，但运行时内核仍偏集中，执行节点、上下文构建、失败恢复和工具治理还未完全平台化。

本轮目标是吸收 `claw code` 的平台化思路，在不破坏当前前端工作台体验的前提下，完成运行时主骨架的第一轮重构。

## 2. Goals

- 统一唯一执行节点实现
- 拆出上下文构建器
- 引入执行节点注册表
- 拆出失败恢复策略模块
- 为工具池和引用映射预留标准接口

## 3. User Stories

### US-R1：统一唯一执行节点实现

作为开发者，我希望后端只保留一套执行节点构建逻辑，这样前端和测试可以依赖唯一主实现。

验收标准：

- 只保留一套执行节点主构建逻辑
- `TaskResponse` 继续输出统一 `execution_nodes`
- 删除双轨节点实现的歧义
- 补齐对应单元测试

### US-R2：拆出 ContextBuilder

作为开发者，我希望上下文分层构建从 `AgentService` 中独立出来，这样上下文预算、压缩和来源说明可以单独治理。

验收标准：

- 新增独立 `ContextBuilder`
- `runtime_agent.py` 改为调用 builder
- 保持现有 `context_layers` 输出兼容
- 增加针对 builder 的最小测试

### US-R3：引入 ExecutionRegistry

作为开发者，我希望关键执行节点拥有统一元数据注册表，这样执行、恢复和前端展示可以共享同一套定义。

验收标准：

- 新增 `ExecutionRegistry`
- 关键节点至少包含 kind、title、可视化标签和恢复元数据
- 主链路接入 registry
- 前端现有页面继续可用

### US-R4：拆出 RecoveryPolicy

作为系统维护者，我希望失败恢复决策从编排器中拆出来，这样重试、跳过失败步骤和重规划策略更易维护。

验收标准：

- 新增独立恢复策略模块
- 保持现有恢复行为兼容
- 增加针对恢复决策的单元测试
- 不破坏当前任务恢复接口

### US-R5：为 ToolPool 和 citation map 预留接口

作为平台演进维护者，我希望工具与引用映射具备统一扩展入口，这样后续可以继续向插件化和后端标准引用推进。

验收标准：

- 工具元数据接口具备类别、权限和展示信息
- citation map 有清晰的后端接口预留
- 当前功能不回退
- 文档说明下一步扩展方式
